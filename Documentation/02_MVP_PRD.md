# MVP PRD（要件定義）：案件管理＋写真/書類アップロード

## 0. このPRDの目的
MVP（最小限の実現可能なモデル）として、案件単位で「写真・PDF・ステータス」を一元管理し、`LINE/メール/多段転送`を減らす。

## 1. 用語
- `案件（Case）`：管理番号（案件ID）単位で紐づく作業/証憑の塊
- `管理者`：元請け側（全案件閲覧/管理）
- `利用者`：ディーラー/法人側（自社が関わる案件の作成・アップロード中心）
- `添付`：PDF/画像などのファイル
- `途中写真`：複数枚の進捗画像（時系列閲覧を重視）

## 2. ユーザーストーリー（最小）
1. 管理者は、案件一覧から必要案件を開き、添付（写真/PDF）とステータスを確認できる
2. 利用者は、案件を作成し、管理番号に紐づく形で写真やPDFをアップロードできる
3. 閲覧者（管理者/利用者）は、必要な情報にすぐ到達できる（探し回らない）
4. 写真は劣化しにくい品質で保存され、端末（スマホ/PC）で閲覧できる

## 3. MVPスコープ（含むもの）
### 3.1 認証・アカウント
- ログイン/ログアウト
- 管理者：全閲覧・管理
- 利用者：所属（ディーラー/法人）に紐づく案件のみ作成・アップロード・閲覧

### 3.2 案件管理
- 案件の作成（利用者が新規依頼を登録）
- 案件一覧（ステータス、案件番号、関連先名など最低限）
- 案件詳細（案件ID単位の「1画面完結」寄りUI）

### 3.3 添付（写真・PDF）
- 画像のアップロード（複数枚）
- PDF/書類のアップロード（見積/納品書などを想定した所定欄）
- 添付の一覧表示（サムネイル＋クリックで拡大/プレビュー）
- 添付の紐付け（どの「欄」に入れたかが分かること）

### 3.4 ステータス更新（最小）
- ステータス（例：受付/確認中/見積済/作業中/納品済/完了）を案件に紐づけ
- 管理者がステータスを更新できる（必要に応じて利用者側も限定的に更新可）

### 3.5 モバイル対応
- Webでスマホ撮影→アップロード→登録が可能
- 最小操作で「次に何をすべきか」が分かる導線

## 4. MVPスコープ外（今はやらない）
- 複雑な原価/会計/請求書の自動起票（PDF添付まではMVP）
- 部品発注書の連携（印刷/外部ソフト前提）
- 既存見積ソフト/専用ソフトとの深い統合
- 1対複数/複数対複数の完全な関係者ロール（将来拡張）

## 5. データモデル（最低限）
- `Organization`（ディーラー/法人）
- `User`（ログインユーザ）
- `Case`（案件：案件ID、顧客/関連先、ステータス、日時）
- `Attachment`（添付：種類（見積/納品/途中写真/その他）、ファイルURL、アップロード日時）

## 6. 画面要件（最低セット）
### 6.1 ログイン画面
- ID/パスワード（またはメールリンク方式：要協議）

### 6.2 管理者：案件一覧
- 検索/絞り込み（最低限：ステータス、案件ID、関連先）

### 6.3 管理者：案件詳細（1画面のイメージ）
- 案件ヘッダ（案件ID、関連先、ステータス）
- 所定欄：見積PDF / 納品書PDF / その他PDF（必要最小）
- 途中写真：サムネイル一覧＋クリックで拡大
- ステータス更新ボタン（管理者）

### 6.4 利用者：新規案件作成
- 必須入力（顧客/車両などはMVPで最低限に制限）

### 6.5 利用者：案件詳細（アップロード中心）
- 写真/書類アップロードUI（追加→保存）
- 自社案件の閲覧

## 7. アクセプタンス基準（受け入れ）
- 利用者は、スマホで写真とPDFをアップロードし、管理者が同じ内容を閲覧できる
- 閲覧者が案件詳細で必要情報（ステータス、見積/納品、途中写真）に迷わず到達できる
- 添付の誤紐付けが起きにくい（欄が明確で、アップロード種類が固定/選択できる）
- ログインにより権限制御が成立している（他社案件が見えない）

## 8. 初期ステータス例（MVP）
- 受付
- 確認中
- 見積済
- 作業中
- 納品済
- 完了

## 9. 今後の拡張設計（前提）
- 将来、複数対複数（例：保険会社/他関係者が同一案件を参照）のために、関係者テーブル/権限制御を拡張可能な形にする
