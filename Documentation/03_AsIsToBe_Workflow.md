# 業務フロー（As-Is / To-Be）

## 1. 現状フロー（As-Is：改善前）
1. ディーラー担当者から電話・LINEで修理依頼を受ける
2. 見積書を作成し、LINEで送付（写真/情報の追加もLINEで往復）
3. 車両引き取り後、元請け側で全体写真・傷の確認写真を撮影
4. 作業指示書を作成して現場へ伝達
5. 現場での途中写真をLINE等で回収
6. 元請けへ転送（LINE/メールで多段共有）
7. 納品書を渡して完了（請求書等も別経路でやり取り）
8. 各段階の情報がフォルダ/ファイル/ツールに分散し、探す・確認する手間が増える

## 2. 理想フロー（To-Be：改善後）
前提：取引相手（ディーラー/法人等）ごとにログインアカウントを用意し、`案件ID`単位で管理する。

### 2.1 受付〜案件作成
1. 利用者（ディーラー/法人）がログイン
2. 「新規依頼」を作成し、案件ID（管理番号）が発行される
3. 所定欄（見積/納品/途中写真 等）を含む案件ページが自動で生成される

### 2.2 写真・書類の登録
1. 利用者が案件ページへ写真/必要書類（PDF）をアップロード
2. アップロードされた添付は、案件詳細ページ上で閲覧可能になる
3. 閲覧者（管理者/利用者）は、案件ページで「必要情報がまとまっている」状態を確認できる

### 2.3 ステータスと進捗共有
1. 管理者は案件ページでステータスを更新し、進捗を可視化する
2. 途中写真（複数枚）は時系列で積み上がり、転送作業が不要になる

### 2.4 完了〜納品/請求の準備
1. 納品書PDF等の添付が追加され、完了判断に必要な証跡が案件内に揃う
2. 必要に応じて請求書等は（MVPでは）添付または外部運用を許容し、段階的に統合する

## 3. 漏れを防ぐ設計ポイント（MVP）
- 「案件ID」単位で所定欄を固定し、誤った添付を混ぜにくくする
- 写真はアップロード種類を明確に（例：全体/傷/途中/納品 等）
- 閲覧権限により他社/他案件を見えないようにする
