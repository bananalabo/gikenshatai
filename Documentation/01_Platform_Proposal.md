# 企画書（案）：B2B 車両修理 証憑/写真/書類一元プラットフォーム

## 1. 背景・課題（現状）
法人取引（B2B）の窓口・情報伝達が `LINE` / `メール` / `電話` などに分散しており、以下が多段・煩雑になっている。

- 見積書 / 納品書 / 請求書 / 交換部品に関する書類のやり取り
- 写真（全体写真・傷確認・途中写真など）の伝送・再共有
- 現場からの途中写真を、元請けへ転送する作業が発生
- 管理者（元請け側）が各工程で介在し、情報が俗人化する
- 関係者ごと（ディーラー/法人/保険等）に手順がバラけるため、漏れのリスクが高い

## 2. 理想像（To-Be）
取引の案件（`管理番号`単位）に紐づく情報を、統一されたプラットフォーム上で一元管理する。

- 受付段階：依頼者側（ディーラー/法人）が写真・必要書類をアップロード
- 管理番号（案件ID）ごとにフォルダ/ページが自動生成され、関係者が閲覧可能
- 見積書（PDF）/ 納品書等（PDF/添付）を案件内の所定欄へ格納
- 途中写真（複数）を案件内へ蓄積し、閲覧者（元請け/関係者）が時系列で確認できる
- ステータス更新により進捗が把握でき、情報転送の多段動作を減らす
- 新しい営業担当者でも閲覧でき、属人性（俗人化）を低減する

## 3. 想定ユースケース（例：元請け＋ディーラー）
1. ディーラー側がログインして新規依頼を作成（管理番号発行）
2. ディーラーが見積に必要な写真/書類（アップロード）を登録
3. 元請けが案件を確認し、必要に応じてPDF等の格納を進める
4. 作業開始後、途中写真が案件内へ蓄積され、元請けへ転送不要にする
5. 納品書・完了情報も案件内で確認でき、完了報告までを完結させる

## 4. MVP方針（最小限の実現可能な範囲）
MVPでは「全機能」を狙わず、まずは以下の成立性（漏れ防止・情報伝達のタイムリー化）を最短で検証する。

- 案件管理（管理番号単位）
- 依頼受付〜ステータス更新（最小）
- 写真/ファイルのアップロードと閲覧（モバイル対応Web）
- PDF（見積・納品書など）の添付/差し込み（最小UI）
- 閲覧権限（管理者＝元請け / 利用者＝ディーラー/法人、最低限）

## 5. 管理者・利用者（アカウント）イメージ
- 管理者：元請け（社内管理者）＝全案件閲覧/管理
- 利用者：ディーラー/法人＝自社が関わる案件の作成・アップロード・閲覧（MVP）
- 将来：複数対複数（例：保険/複数関係者が同一案件を参照）へ拡張可能な設計方針

## 6. 技術・運用上の論点（MVPの意思決定）
写真・画像の品質と閲覧体験を重視するため、アップロード先は以下を比較し意思決定する。

- 自前サーバー：UI/UXと品質を確保しやすいがコスト増の可能性
- Google Drive：コストは抑えやすいがUI/UXが劣る可能性

## 7. 目標（コスト感）
- ランニング費用目標：月 `3〜5万円`（MVPで成立させたい）
- 参考例：既存類似では月 `5.5万円` 程度のランニングがある（写真/機能過多だが一例）
- 写真品質は劣化させない前提で検討（画像の荒さは不可）
