# 権限設計（Roles & Permissions）

## 1. 目的
- `漏れ`と`見え過ぎ`を同時に防ぐ（俗人化の原因を減らす）
- 少なくともMVPでは「利用者が自社案件のみを操作/閲覧できる」状態を作る

## 2. ロール定義（MVP）
### 2.1 管理者（元請け）
- 全案件一覧の閲覧
- 全案件詳細の閲覧
- ステータス更新
- 添付の閲覧（全種類）
- 必要に応じて、添付の追加/更新（MVPでは運用を決める）

### 2.2 利用者（ディーラー/法人）
- 自社に紐づく案件のみ作成
- 自社に紐づく案件の詳細閲覧
- 写真/書類（PDF等）のアップロード
- 必要に応じて、限定された範囲でステータス入力（MVPでは「管理者のみ更新」を基本とするのが安全）

## 3. アクセス制御モデル（最小）
- `User` は必ず `Organization` に紐づく
- `Case` は必ず `Organization`（作成元/担当）を持つ
- 権限制御ルール（MVP）：
  - 利用者は `Case.organization_id == user.organization_id` の案件のみアクセス可
  - 管理者は全案件アクセス可

## 4. 添付（Attachment）の権限
- 添付は「案件に紐づく」ので、案件閲覧権限と同じ範囲で閲覧可能
- 利用者はアップロードのみ（削除/編集の扱いは運用で決める）
- 管理者は必要に応じて編集/削除可能（MVPでの運用方針を開発時に確定）

## 5. 関係者拡張（将来の前提）
- 将来「複数対複数」になるため、現段階では次を前提に設計しておく
  - `Case` と `Viewer`（閲覧者）を分離できるデータ設計にしておく
  - 将来の関係者追加時に「権限付与」で対応できるようにする

## 6. 受け入れ条件（セキュリティ）
- 利用者が他社の案件IDを知ってもアクセスできない
- URL直打ちで他社案件が見えない（認可でブロック）
